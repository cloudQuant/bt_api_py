### 背景

binance、okx、ib、ctp 这几个交易所的接入与测试已完成并稳定运行。后续新增的其他交易所（如 bybit、kraken、mexc、huobi/htx 等）在接口行为与测试覆盖上存在差异，导致回归测试不通过。

### 目标

- 让所有新增交易所的接口行为与 binance/okx 的既定规范保持一致（入参、返回字段、错误码、精度/最小单位、节流策略）。
- 修复或补齐对应的单元/集成测试，确保 `pytest` 全量运行通过。

### 范围

- 代码层面：交易所适配器、签名/鉴权、时间戳转换、订单/行情/账户等核心 API 的数据规范化。
- 测试层面：`tests/` 下各交易所的单测与集成测试，以及根目录下的 `test_*.py` 文件。
- 不在范围：已稳定的 binance、okx、ib、ctp 的核心逻辑（除非为统一接口所需的最小改动）。

### 交付标准（验收）

- 本地执行 `pytest -q`，所有测试通过（含新增或调整的用例）。
- 新增或修改的行为差异在测试中有明确断言；必要时为精度/节流/错误码差异添加回归用例。
- 更新相关文档与示例（如 README/IMPROVEMENTS 需要引用时），说明接口差异与统一规范。

### 修复原则（优先级从高到低）

1. **源代码优先**：遇到测试失败时，首先判断是否是源代码（`bt_api_py/` 下的业务代码）存在 bug。若源代码有误，修复源代码而非修改测试来迁就错误行为。
2. **参照 binance/okx 标准**：binance 和 okx 的源代码与测试已经稳定运行，是所有新增交易所的参照基准。新增交易所的源代码实现（适配器、签名、数据规范化）应与 binance/okx 保持一致的模式和接口契约。
3. **测试用例其次**：仅当确认源代码实现正确后，才修改测试用例。测试用例的修改范围包括：
   - 修复测试代码本身的逻辑 bug（如变量赋值错误、断言条件错误）。
   - 更新断言值以匹配 YAML 配置中的正确值（如 `exchange_name`）。
   - 为需要网络的测试添加 `@pytest.mark.integration` 标记。
   - 为不存在的方法添加 `hasattr` 防御性检查。
4. **不可削弱测试**：不允许通过删除断言、降低覆盖率或注释掉测试来"修复"失败。

### 具体排查步骤（每个交易所）

1. 运行该交易所的测试文件：`pytest tests/feeds/test_<exchange>.py -q --tb=short`。
2. 对每个失败的测试用例，**先查看对应的源代码**（`bt_api_py/feeds/live_<exchange>/`、`bt_api_py/containers/exchanges/<exchange>_exchange_data.py`），判断失败原因：
   - **源代码 bug**（如 `_get_tick` 方法既调用 `self.request()` 又返回给上层再次调用 `self.request()`，造成双重请求）→ 修复源代码。
   - **配置加载 bug**（如 `base_urls.rest` 是 dict 但直接赋值给 `self.rest_url`，应提取对应 asset_type 的 URL 字符串）→ 修复源代码。
   - **缺失属性**（如 exchange_data 缺少 `api_key`/`api_secret` 属性导致 `AttributeError`）→ 修复源代码。
   - **测试代码 bug**（如 `data = None` 无条件覆盖、`if ...: pass` 未保护后续代码）→ 修复测试。
   - **断言值过时**（如 YAML 中 `exchange_name: xxxSpot` 但测试断言 `== "xxx"`）→ 修复测试。
   - **网络依赖**（测试调用真实 API 但未标记 `@pytest.mark.integration`）→ 添加标记。
3. 修复后重新运行该交易所测试，确认全部通过。
4. 进入下一个交易所。

### 参照模板

- **源代码参照**：`bt_api_py/feeds/live_binance/`、`bt_api_py/containers/exchanges/binance_exchange_data.py`
- **测试参照**：`tests/feeds/test_binance.py`、`tests/containers/exchanges/test_binance_wallet_api.py`
- **配置参照**：`bt_api_py/configs/binance.yaml`

### 工作建议

- 先对比 binance/okx 的接口期望（代码 + 测试），列出各交易所的差异清单。
- 优先修复源代码中的 bug，再处理测试层面的问题。
- 对外部依赖的接口（真实交易所）可使用录制的 mock/fixture，减少波动。
- 变更较大的模块提交前跑一次定向测试：`pytest tests/feeds/test_<exchange>.py -q`。

### 限制

1. 不允许一次性进行全部交易所的测试，耗费时间太久，一次最多测试一个交易所。即不允许pytest tests 这种整个测试文件夹所有的测试用例一次运行的。
2. 当我取消掉运行了，就证明你代码不对，需要重新考虑运行的代码再继续重新运行。
