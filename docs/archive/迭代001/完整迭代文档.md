## 2. 初始需求复述

### 2.1 原始背景

当前 `bt_api_py` 已经相对成熟，但由于接入了大量交易所，仓库逐步变得臃肿。项目希望只保留核心能力，把交易所相关内容拆分到独立 Python 包中，并在安装这些子包后，依然可以用和当前类似的方式调用。

### 2.2 原始任务

初始需求明确提出，至少将以下能力从主包中拆分出去：

- `ctp`
- `ib_web`
- `mt5`
- `binance`
- `okx`

目标产物形式示例：

- `bt_api_ctp`
- `bt_api_ib_web`
- `bt_api_mt5`
- `bt_api_binance`
- `bt_api_okx`

并统一放入一个独立目录中，后续可分别打包发布。

## 3. 本轮迭代为什么必须做

### 3.1 业务层原因

随着交易所数量增加，`bt_api_py` 已经不仅是“核心交易 API”，而更像“核心框架 + 多交易所实现全集”。这种结构在早期有利于快速扩展，但在当前阶段会带来明显问题：

- 主包安装体积持续增大
- 不同交易所依赖差异越来越大
- 原生扩展、浏览器依赖、桌面交易软件依赖混入同一个包
- 发布一个交易所的小改动，也可能触发整个主包重新发布
- 维护者难以判断某次改动影响范围

### 3.2 工程层原因

结合当前仓库现状，可以确认以下事实：

1. `bt_api_py/bt_api.py` 当前通过自动扫描 `bt_api_py.exchange_registers` 的方式完成交易所注册。
2. `bt_api_py/gateway/runtime.py` 当前把 `CTP / IB_WEB / BINANCE / OKX / MT5` 适配器直接内置在运行时注册表里。
3. `setup.py` 当前仍直接打包 `CTP` 原生扩展和运行时库文件。
4. `pyproject.toml` 虽然已经存在 `ib_web`、`mt5` 等 optional dependencies，但代码组织仍然是单体仓库内聚合。

这说明当前系统虽然已经具备“注册式扩展”的雏形，但还没有真正具备“外部插件包可插拔”的结构。

### 3.3 当前单体结构的直接问题

- **安装成本高**：用户只想用 `binance`，却要一起携带与 `ctp`、`ib_web`、`mt5` 相关的代码和构建负担。
- **发布耦合高**：一个交易所修改，很容易变成整个主包版本升级。
- **依赖冲突风险高**：`playwright`、原生库、桌面软件桥接等依赖并不适合和纯 Python HTTP 接口强绑定。
- **扩展边界不清**：核心框架职责与交易所实现职责交叉，后续接入新交易所的成本会上升。
- **测试成本高**：不同交易所测试条件差异大，CI 和本地验证越来越难统一。

## 4. 本轮迭代目标

## 4.1 一句话目标

让 `bt_api_py` 首次具备**外部插件发现、注册与加载能力**，并以 `Binance / OKX` 两个纯 Python 交易所插件完成端到端试点验证。

## 4.2 迭代001正式交付目标

### 目标A：完成核心插件化能力建设

主包 `bt_api_py` 在本轮需要完成：

- 插件发现与加载机制
- 面向 `ExchangeRegistry` 的统一注册协议
- 面向 `GatewayRuntime` 的外部适配器注册能力
- 插件加载日志、失败隔离与版本兼容检查

### 目标B：完成第一批试点插件验证

本轮正式交付的试点对象为：

- `bt_api_binance`
- `bt_api_okx`

两者的目标不是一次性覆盖所有历史能力，而是验证“拆出去之后仍可被核心正常发现、注册、调用和测试”。

### 目标C：保证迁移期兼容

在迁移期内，已有调用方式应尽量少改，至少保证：

- 安装对应插件后，`BtApi` 能正常识别对应交易所
- 现有 `ExchangeRegistry` 路由能力可继续使用
- `GatewayRuntime` 可以通过插件注册适配器，而不是持续硬编码在主包里
- 老项目在兼容窗口内不被强制要求立即重写业务调用

## 4.3 专项总体目标（后续迭代承接）

本次迭代之外，专项的整体目标仍包括：

- 后续拆分 `bt_api_ib_web`
- 后续拆分 `bt_api_mt5`
- 单独波次处理 `bt_api_ctp`
- 最终把主包收敛为真正的“核心框架包”

这些目标在本文档中会保留路线图描述，但**不作为迭代001的全部承诺项**。

## 5. 非目标

本轮明确**不做**以下事项：

1. 不一次性拆分全部交易所。
2. 不重写 `BtApi` 全部公共接口。
3. 不在本轮完成所有历史测试债清理。
4. 不把所有 exchange-specific container 全部重构成全新模型。
5. 不在没有必要的情况下改动已有用户调用参数和业务语义。
6. 不将“拆包”扩展成“全面架构重写”。

## 7. 当前现状分析

## 7.1 当前核心入口现状

`bt_api_py/bt_api.py` 当前在导入时会扫描 `bt_api_py.exchange_registers` 下的模块并导入，从而触发注册逻辑。这意味着：

- 当前“交易所接入”本质上已经依赖注册表机制
- 但注册模块必须存在于 `bt_api_py` 主包内部
- 外部包目前还无法自然接入这个扫描流程

换句话说，系统已经具备“插件化的形”，但还缺“插件包化的神”。

## 7.2 当前网关运行时现状

`bt_api_py/gateway/runtime.py` 中的 `ADAPTER_REGISTRY` 直接内置：

- `CTP`
- `IB_WEB`
- `BINANCE`
- `OKX`
- `MT5`

这说明网关层虽然允许 `register_adapter()` 扩展，但默认实现仍然和具体交易所耦合在主包里。因此如果要真正拆包，网关适配器也要从“内置”切换到“插件注册”。

## 7.3 当前打包现状

### CTP

`setup.py` 当前直接定义了 `bt_api_py.ctp._ctp` 扩展和对应 runtime library 打包逻辑。CTP 是本轮里风险最高、拆分最不适合直接用纯 Python 模式处理的对象。

### IB Web / MT5

`pyproject.toml` 已经存在 `ib_web`、`mt5` optional dependencies，这说明项目已经感知到它们不应当成为所有用户的默认依赖。

### Binance / OKX

Binance 与 OKX 当前主要以纯 Python 请求层、WebSocket 流、交易所数据结构和注册器形式存在，更适合作为第一批插件化试点。

## 7.4 当前架构结论

当前系统不是“完全不可拆”，而是：

- **注册中心已经存在**
- **自动发现机制还只支持主包内部模块**
- **网关适配器仍有内置耦合**
- **CTP 打包链路需要特殊处理**

因此本轮最合理的策略不是直接暴力搬目录，而是先补齐插件边界，再按批次迁移交易所实现。

## 8. 目标架构

## 8.1 目标架构原则

本轮拆包后，整体架构应遵循以下原则：

1. **核心最小化**：主包只保留跨交易所共享的能力。
2. **插件自包含**：每个交易所包自己携带实现、注册逻辑和必要依赖。
3. **调用稳定**：用户安装插件后，尽量继续通过 `BtApi` 统一调用。
4. **发现解耦**：核心不再写死每个交易所模块路径。
5. **版本可控**：插件与核心之间要有明确兼容关系。

## 8.2 目标包划分

### 主包：`bt_api_py`

职责建议保留：

- `BtApi`
- `ExchangeRegistry`
- 核心异常和错误类型
- 通用容器基类
- 通用请求客户端、连接基础设施
- 通用网关协议与运行时框架
- 插件加载器
- 插件注册协议

### 插件包：`bt_api_binance`

职责建议包含：

- Binance request/feed 实现
- Binance WebSocket 实现
- Binance exchange data
- Binance register 模块
- Binance gateway adapter
- Binance 插件入口

### 插件包：`bt_api_okx`

职责建议包含：

- OKX request/feed 实现
- OKX WebSocket 实现
- OKX exchange data
- OKX register 模块
- OKX gateway adapter
- OKX 插件入口

### 插件包：`bt_api_ib_web`

职责建议包含：

- IB Web request/feed 实现
- IB Web stream 实现
- IB Web session 辅助函数
- IB Web exchange data
- IB Web register 模块
- IB Web gateway adapter
- IB Web 插件入口

### 插件包：`bt_api_mt5`

职责建议包含：

- MT5 适配实现
- MT5 gateway adapter
- MT5 register 模块
- MT5 插件入口

### 插件包：`bt_api_ctp`

职责建议包含：

- CTP 原生扩展构建逻辑
- CTP feed / stream / register
- CTP exchange data
- CTP gateway adapter
- CTP 插件入口
- CTP runtime libraries / framework files

## 8.3 推荐插件发现机制

本轮建议采用“**入口点 + 显式注册**”的方式，而不是继续依赖扫描主包内部目录。

建议机制：

1. 每个插件包在安装后暴露统一入口，例如 `register_plugin()`。
2. 核心包在启动时通过 `importlib.metadata.entry_points()` 发现已安装插件。
3. 插件入口负责向 `ExchangeRegistry` 和 `GatewayRuntime` 注册自身能力。
4. 核心包保留兼容兜底逻辑，但不再要求插件代码必须放在 `bt_api_py.exchange_registers` 下。

## 8.4 为什么不建议继续使用“扫描主包目录”方案

如果仍然要求插件把模块塞回 `bt_api_py.exchange_registers`：

- 主包命名空间会继续膨胀
- 外部包边界会非常模糊
- 发布与维护难度会重新回到单体模式
- 对 PyPI 分发和多包版本治理不友好

因此，本轮必须把“扫描内部目录”升级为“扫描已安装插件入口”。

## 9. 拆包边界设计

## 9.1 保留在主包中的内容

以下内容建议保留在 `bt_api_py`：

- 统一异常类型
- 核心日志、事件总线、注册中心
- 通用 `BtApi` 路由逻辑
- 通用 gateway runtime / protocol / health / subscription manager
- 与具体交易所无关的通用 containers 基类
- 与具体交易所无关的工具函数
- 插件协议与兼容检查器

## 9.2 必须迁移到插件包中的内容

以下内容应随插件迁移：

- 交易所专用 request/feed 类
- 交易所专用 WebSocket / stream 实现
- 交易所专用 gateway adapter
- 交易所专用 register 模块
- 交易所专用 exchange data
- 交易所专用鉴权、session、配置工具
- 该交易所独有测试
- 该交易所独有依赖

## 9.3 边界判断规则

若模块满足以下任一条件，应优先归入插件包：

1. 模块名称或职责明显绑定某个交易所。
2. 模块依赖该交易所专有鉴权、协议或 SDK。
3. 模块只会被一个交易所使用。
4. 模块改动时不应影响其他交易所。

若模块满足以下条件，则应保留在主包：

1. 多交易所共用。
2. 与交易所品牌无关。
3. 是所有插件都需要依赖的基础能力。
4. 属于统一抽象、协议或运行时框架。

## 10. 专项拆分包优先级清单

| 包名 | 当前优先级 | 风险等级 | 拆分建议 | 说明 |
|---|---:|---:|---|---|
| `bt_api_binance` | P0 | 中 | 第一批试点 | 纯 Python 能力较多，适合作为插件化样板 |
| `bt_api_okx` | P0 | 中 | 第一批试点 | 结构相对清晰，适合与 Binance 共同验证方案 |
| `bt_api_ib_web` | P1 | 中高 | 第二批 | 依赖 Web 会话、浏览器或认证链路，复杂度高于纯 REST/WSS |
| `bt_api_mt5` | P1 | 高 | 第二批 | 依赖本地终端环境，不适合作为第一批样板 |
| `bt_api_ctp` | P0 | 很高 | 单独波次处理 | 涉及原生扩展和运行时库，需独立验证打包链路 |

说明：

- 上表是整个拆包专项的优先级视图，不等于“迭代001全部交付清单”。
- 迭代001正式交付对象仍以 `bt_api_binance`、`bt_api_okx` 为准。
- `bt_api_ctp` 虽然战略优先级高，但因风险高，不应挤入试点波次。


每个子包推荐结构：

```text

  README.md
  src/
    bt_api_binance/
      __init__.py
      plugin.py
      feeds/
      gateway/
      exchange_data/
      register.py
  tests/
```

说明：

- 迭代001至少需要真正落地 `bt_api_binance`、`bt_api_okx` 两个目录。
- `bt_api_ib_web`、`bt_api_mt5`、`bt_api_ctp` 目录可以作为规划位保留，但不要求在本轮全部创建并发布。
- 插件包内部不必强行复刻 `bt_api_py` 全部目录层级。
- 插件包只需满足“自己可以实现并注册能力”的目标。
- 如果为了迁移成本控制，第一阶段临时沿用部分旧目录命名，也可以接受，但最终应收敛为插件自洽结构。

## 12. 统一插件协议设计

## 12.1 插件注册接口

每个插件包建议提供统一注册入口，例如：

- `register_plugin()`
- 或 `setup_plugin(registry, gateway_runtime)`

注册动作至少覆盖：

- `ExchangeRegistry.register_feed(...)`
- `ExchangeRegistry.register_exchange_data(...)`
- `ExchangeRegistry.register_balance_handler(...)`
- `ExchangeRegistry.register_stream(...)`
- `GatewayRuntime.register_adapter(...)`

## 12.2 插件元信息

每个插件应声明：

- 插件名称
- 支持的交易所编码
- 支持的资产类型
- 依赖的核心版本范围
- 插件自身版本

## 12.3 核心兼容检查

主包加载插件时建议进行版本兼容校验：

- 若插件要求的核心版本不匹配，应给出明确错误
- 若插件注册不完整，应给出明确日志和失败原因
- 插件加载失败不应拖垮整个主包启动，但要有可观测日志

## 13. 迁移策略

## 13.1 总体策略

本轮采用“**先打通插件机制，再分批迁移交易所**”的策略，而不是直接拆目录。

推荐顺序：

1. 先改主包，让它支持发现和加载外部插件。
2. 再挑两个相对标准的交易所做试点。
3. 再处理依赖更重、环境更复杂的对象。
4. 最后收尾清理主包中的历史残留实现。

## 13.2 兼容期策略

建议至少保留一个迁移期版本：

- 主包仍可兼容旧的内部注册方式
- 新插件方式可同时存在
- 文档明确标记内部注册为逐步废弃路径

这样可以降低一次性大迁移风险。

## 13.3 用户迁移体验目标

用户侧预期体验应为：

```bash
pip install bt_api_py
pip install bt_api_binance
```

然后即可继续使用类似：

- `BtApi({...})`
- `get_tick(...)`
- `make_order(...)`
- `subscribe(...)`

也就是说，用户新增成本应主要体现为“多安装一个包”，而不是“重学一套新 API”。

## 13.4 本轮最小闭环

迭代001必须形成一个最小且可演示的业务闭环：

1. 在干净环境中安装 `bt_api_py` 时，不强制引入 `IB Web`、`MT5`、`CTP` 相关重依赖。
2. 额外安装 `bt_api_binance` 或 `bt_api_okx` 后，核心可自动发现并加载插件。
3. 用户继续使用现有主入口，例如 `BtApi({...})`，即可访问试点交易所能力。
4. `GatewayRuntime` 可以通过插件完成适配器注册，而不是继续依赖核心硬编码。
5. 插件缺失、版本不兼容或加载失败时，系统报错清晰且不会拖垮整个主包启动。

只要上述闭环没有形成，即使文档、目录或样板包已经存在，也不能视为本轮成功。

## 13.5 产品成功指标

从产品角度，本轮至少要满足以下成功标准：

1. **安装体验收敛**：只安装 `bt_api_py` 时，不应强绑定浏览器类、桌面终端类或 CTP 原生扩展依赖。
2. **迁移成本可控**：老用户迁移到试点插件模式时，新增动作应尽量控制为“补装插件包”，而不是重写主要业务调用。
3. **发布边界清晰**：完成试点后，Binance / OKX 的后续演进可以主要在插件包维度推进，而不是继续要求核心包频繁承载其实现变更。
4. **扩展路径成立**：团队新增一个同类交易所插件时，不需要重复修改核心入口或网关运行时硬编码。

## 17. 验收标准

## 17.1 架构验收标准

满足以下条件，才算迭代001落地成功：

1. `bt_api_py` 已具备外部插件发现、注册与加载能力。

3. `GatewayRuntime` 支持通过插件注册适配器，而不是继续依赖新增硬编码。

5. 在兼容窗口内，旧扫描路径仍可兜底工作，但新插件路径已成为主推荐方案。

## 17.2 功能验收标准

对本轮已拆出的试点插件包，至少满足：

1. 能正常安装。
2. 能完成插件注册。
3. 能通过最小可运行示例。
4. 能通过对应测试集。
5. 缺失依赖或版本不匹配时，报错明确。

1. `bt_api_binance`、`bt_api_okx` 具备独立 `pyproject.toml`。

3. 核心与试点插件之间兼容关系明确可查。
4. 本轮不要求所有后续插件同步达到发布状态。

## 17.4 文档验收标准

1. 用户可以根据文档完成安装。
2. 开发者可以根据文档新增一个同类插件包。
3. 迁移说明能回答“老用户怎么升级”这个问题。

## 17.5 产品验收标准

1. 用户升级到试点方案时，新增复杂度主要表现为“多安装插件包”，而不是学习全新 API。

## 19. 风险清单

## 19.1 高风险项

### 风险1：CTP 原生扩展拆分失败

影响：高
说明：CTP 当前已经深度绑定在主包构建链路中，单独拆分时最容易出现编译、运行时库路径和平台兼容问题。

应对：

- CTP 独立为单独波次
- 优先验证构建链路，再验证业务功能
- 不把 CTP 作为第一批插件样板

### 风险2：核心仍残留对具体交易所的隐式依赖

影响：高
说明：如果 `BtApi`、`GatewayRuntime`、工具函数中仍散落具体交易所导入，拆包后会产生隐藏 ImportError。

应对：

- 在拆分前做依赖扫描
- 明确禁止核心层直接 import 插件实现
- 增加“无插件安装环境”的验证用例

### 风险3：插件发现机制设计不稳

影响：高
说明：如果插件发现方案过于依赖路径或命名约定，后续扩展会再次回到单体耦合。

应对：

- 采用标准入口点机制
- 对插件元信息和兼容规则做强约束

## 19.2 中风险项

### 风险4：用户迁移成本过高

影响：中
说明：如果要求用户同时修改太多调用代码，会导致升级阻力大。

应对：

- 优先保持 `BtApi` 调用方式不变
- 用“新增安装插件”替代“重学接口”

### 风险5：测试边界被打散

影响：中
说明：拆包后如果测试没有同步拆分，CI 和本地验证会变得混乱。

应对：

- 插件包带走自身测试
- 核心仓库保留最小兼容集成验证

## 23. 关键决策点

建议在迭代001评审时，直接锁定以下决策，避免边做边讨论：

| 决策项 | 当前建议结论 | 原因 | 是否纳入迭代001 |
|---|---|---|---|
| 子包托管方式 | 先放在单仓库 `packages/` 下管理 | 先降低治理成本，优先验证插件机制 | 是 |
| 插件发现机制 | 采用 `entry_points` 作为主方案 | 标准化、可分发、可扩展 | 是 |
| container 迁移边界 | 本轮优先迁出 exchange data / feed / stream / adapter，暂不强求所有专用 container 一次性迁净 | 控制范围，先跑通插件闭环 | 是 |
| CTP 上线节奏 | 允许晚于其他插件一个版本波次上线 | 原生扩展风险最高，不应阻塞试点 | 是 |
| 兼容期策略 | 保留旧扫描逻辑一个兼容窗口版本 | 降低迁移风险，给老用户缓冲 | 是 |

## 24. 最终结论

本次“拆包”不是简单的代码搬家，而是一次针对 `bt_api_py` 产品形态和工程边界的重构型迭代。

如果直接从目录层面硬拆，最终很容易形成多个“看起来分开、实际上仍强耦合”的伪插件包；这会增加维护成本，而不是降低成本。

因此，本轮正确的落地路径应当是：

1. 在迭代001内，先把插件机制、注册协议和网关适配扩展能力打通。
2. 在同一轮内，用 `Binance / OKX` 完成试点验证，形成最小闭环。
3. 将 `IB Web / MT5 / CTP` 明确下沉为后续波次，而不是混进本轮承诺范围。
4. 待试点稳定后，再推进复杂依赖插件和主包收口。

按这个路径推进，本文档就不再只是一个“理想化总蓝图”，而会成为一份真正可执行、可验收、可管理预期的迭代001方案。
