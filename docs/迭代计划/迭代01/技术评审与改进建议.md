## 1. 架构设计评审

### 1.1 bt_api_base 边界定义

**问题**：如何准确划分 bt_api_base 和 bt_api_py 的边界？

**建议**：
- `bt_api_base` = 所有 `bt_api_xx` 插件和 `bt_api_py` 都会用到的代码
- `bt_api_py` = 插件编排层 + gateway + zmq + EventBus
- 判定规则：某个模块如果被**所有**交易所插件用到 → base；如果只被 `bt_api_py` 自身用到 → core

**需要拆分验证的候选**：
- `balance_utils.py`：所有交易所有效 → base
- `rate_limiter.py`：所有交易所用到 → base
- `event_bus.py`：`bt_api_py` 自身使用，插件不直接用 → core
- `containers/orders/`：各交易所订单结构不同，可能是各插件自己持有

### 1.2 插件间隔离

**问题**：bt_api_xx 插件能否相互调用？

**结论**：不允许。插件只能依赖 bt_api_base 和核心 API，不得直接 import 其他插件。

**验证手段**：CI 增加静态检查 `ruff rule: banned-imports`

## 2. CTP 扩展风险评估

### 2.1 复杂度：XL（最高风险）

CTP 是本迭代最复杂的部分，原因是：

| 因素 | 说明 |
|------|------|
| C++ SWIG 扩展 | 需要编译环境，跨平台（macOS/Linux/Windows） |
| 19.8 MB ctp_wrap.cpp | SWIG 生成文件，体积大 |
| 多版本共存 | CTP 6.3/6.5/6.7 等多版本需要兼容 |
| 硬件依赖 | 需要金证柜台或 SimNow 模拟 |

### 2.2 建议

1. **提前 PoC**：在正式开发前，先在干净环境编译 CTP 扩展
2. **CI 三平台**：macOS (x64/arm64)、Linux (manylinux)、Windows 必须覆盖
3. **版本兼容层**：抽离 `OrderRefAllocator` 到纯 Python 层，减少 C++ 耦合
4. **分离发布**：CTP wheel 单独发布，不与其他插件绑定

## 3. bt_api_py 重构评审

### 3.1 兼容性策略

现有用户升级路径：
- 旧用户：`pip install bt_api_py`（含所有交易所）
- 新用户：`pip install bt_api_py bt_api_binance bt_api_okx`（按需）

**DeprecationWarning**：主包对旧路径的 import 给出警告，但不破坏功能。

### 3.2 插件加载顺序

启动流程：
```
1. import bt_api_py
2. PluginLoader.load_all() → 扫描 entry_points
3. 旧路径兼容扫描（可关闭）
```

**风险**：如果用户同时装了插件和旧路径，可能重复注册。

**解决**：GatewayRuntimeRegistrar 同一 exchange_type 重复注册时 warn，不覆盖。

## 4. 测试迁移策略

### 4.1 核心原则

测试必须跟随被测代码一起迁移，不能留在主仓库。

| 场景 | 代码处理 | 测试处理 |
|------|----------|----------|
| 交易所代码迁出 | `git mv` | 同步 `git mv` |
| 某插件完全拆分 | 删除主仓库路径 | 插件包内独立测试 |

### 4.2 验收检查

每个涉及交易所迁移的 PR 必须通过：

```bash
# 主仓库 tests/ 不得残留被迁交易所
grep -rnE "bt_api_py\.feeds\.live_<x>|test_<x>_.*\.py" tests/
# 必须返回 0 条
```
