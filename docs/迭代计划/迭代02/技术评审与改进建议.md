### 1.3 auth_config.py 的交易所特定类问题

当前 `auth_config.py` 包含：
- `CryptoAuthConfig` - 通用加密货币认证（API key + secret）
- `CtpAuthConfig` - CTP 特定（broker_id、md_front、td_front）
- `IbAuthConfig` - IB 特定（host、port、client_id）
- `IbWebAuthConfig` - IB Web 特定（OAuth、cookies）

**建议**：
- `CryptoAuthConfig` 保留在 bt_api_py（通用）
- `CtpAuthConfig` → 迁移到 bt_api_ctp
- `IbAuthConfig` → 迁移到 bt_api_ib（如果存在）
- `IbWebAuthConfig` → 迁移到 bt_api_ib_web
- bt_api_py 只保留 `AuthConfig` 基类和 `CryptoAuthConfig`

## 2. ZMQ 技术选型评审

### 2.1 序列化方案对比

| 方案 | 序列化速度 | 反序列化速度 | 包大小 | 生态 |
|------|-----------|-------------|--------|------|
| msgpack | 极快 | 极快 | 最小 | 好 |
| json | 慢 | 慢 | 大 | 最好 |
| protobuf | 快 | 快 | 小 | 好但需 schema |
| pickle | 快 | 快 | 中 | Python only |

**推荐**：msgpack 作为默认，json 作为调试/兼容模式。

### 2.2 ZMQ Pattern 选择

| 场景 | Pattern | 理由 |
|------|---------|------|
| 行情分发 | PUB/SUB | 一对多广播，topic 过滤 |
| 交易指令 | DEALER/ROUTER | 比 REQ/REP 更灵活，支持异步和多客户端 |
| 心跳监控 | PUB/SUB | 复用行情通道，特殊 topic |

**注意**：REQ/REP 在单个请求未完成时会阻塞，DEALER/ROUTER 更适合高并发场景。建议评估后决定。

### 2.3 高频场景考量

- **消息合并**：同一 symbol 的 ticker 在 HWM 溢出时只保留最新
- **零拷贝**：使用 zmq.Frame 避免不必要的内存拷贝
- **批量发送**：支持 zmq.SNDMORE 多帧消息减少 syscall

## 3. DirectClient 设计评审

### 3.1 同步 vs 异步

当前 bt_api_py 的 BtApi 是同步接口（使用 queue），而现代 Python 趋势是 async。

**建议**：
- DirectClient 统一使用 async 接口
- 提供 `sync_wrapper()` 给不需要 async 的用户
- BtApi 内部使用 event loop 桥接 async DirectClient

### 3.2 错误处理策略

每个交易所的错误码不同，DirectClient 应当：
- 捕获交易所原始错误
- 转换为 bt_api_base 定义的标准异常（RequestError、InvalidOrderError 等）
- 保留原始错误信息在异常的 `detail` 字段中

### 3.3 限速器集成

- DirectClient 应内置限速器
- 限速参数从交易所 YAML 配置读取
- 超过限速时自动排队等待，不抛异常

## 5. CI/CD 建议

### 5.1 Monorepo 构建矩阵

```yaml
# 建议的 GitHub Actions 矩阵
strategy:
  matrix:
    package:
      - bt_api_base
      - bt_api_py
      - bt_api_binance
      - bt_api_okx
      - bt_api_ctp
    python-version: ["3.10", "3.11", "3.12"]
    os: [ubuntu-latest]
```

### 5.2 发布流程

1. bt_api_base 先发布（所有包的依赖）
2. bt_api_xx 可并行发布
3. bt_api_py 最后发布（依赖 bt_api_base 版本号）

### 5.3 版本同步

建议使用 `bumpversion` 或 `hatch-vcs` 统一管理版本号。所有包的大版本号保持一致。

## 6. 性能基线

迭代02完成后应建立的性能基线：

| 指标 | 目标值 | 测量方法 |
|------|--------|----------|
| `import bt_api_py` 耗时 | < 500ms | `time python -c "import bt_api_py"` |
| Plugin 加载（单个） | < 100ms | PluginLoader 内部计时 |
| ZMQ PUB/SUB 吞吐（单线程） | > 100k msg/s | 本地 inproc 测试 |
| ZMQ 延迟（PUB→SUB） | < 1ms (P99) | 本地 tcp 测试 |
| DirectClient REST 调用 | 取决于网络 | 记录基线即可 |
