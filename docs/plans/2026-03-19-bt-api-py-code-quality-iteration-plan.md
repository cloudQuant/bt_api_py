# bt_api_py 代码质量优化迭代计划

## 1. 文档定位

本文件用于作为 `bt_api_py` 当前一轮代码质量治理的执行计划，参考 `backtrader` 仓库“代码质量改进优化”的治理方式，但按 `bt_api_py` 的仓库规模、测试形态和模块风险重新落地。

本计划不是泛泛的长期愿景，而是本轮可执行、可验证、可拆批推进的迭代方案。

与仓库内已有文档的关系如下：

1. `docs/plans/2026-03-16-stability-roadmap.md` 负责描述长期稳定性路线图。
2. `docs/IMPROVEMENT_EXECUTION_PLAN.md`、`CODE_QUALITY.md` 记录历史优化方向和质量工具链。
3. **本文件**负责定义当前一轮代码质量优化的目标、批次、硬门禁、验证方式和交付标准。

后续若要发起 `bt_api_py` 的代码质量优化任务，优先引用本文件。

## 2. 迭代目标

本轮迭代的目标不是“大规模重写”，而是在不扩展业务功能的前提下，优先降低高风险代码的维护成本和回归风险。

核心目标如下：

1. 收敛高风险模块中的广义异常捕获、静默失败和状态不透明问题。
2. 加强 WebSocket、gateway、核心服务层的测试保护和行为可见性。
3. 统一核心 API 和高频调用链的类型标注、文档字符串和异常契约。
4. 通过更现实的质量门禁，让后续每一轮小批次治理都可以持续复用。
5. 在不动 CTP 生成代码和不做全量架构改写的前提下，优先处理最有收益的问题域。

## 3. 当前基线

基于 2026-03-19 对仓库的快速扫描，当前项目具备以下特征：

1. 仓库规模较大，`bt_api_py/` 下约有 `785` 个 Python 文件，`tests/` 下约有 `95` 个测试文件。
2. `pyproject.toml` 已配置 `ruff`、`mypy`、`pytest` 和覆盖率门槛，`coverage fail_under = 80`。
3. 项目已对部分 `requests` 调用补齐 `timeout=`，但异常语义和请求层契约仍不完全一致。
4. WebSocket、gateway、security_compliance、若干 `request_base.py` 中仍存在较多 `except Exception` 或宽泛恢复逻辑。
5. 若干热点模块体量偏大，维护成本高，例如：
   - `bt_api_py/feeds/live_binance/request_base.py`
   - `bt_api_py/websocket/monitoring.py`
   - `bt_api_py/websocket/advanced_connection_manager.py`
   - `bt_api_py/websocket/advanced_websocket_manager.py`
   - `bt_api_py/bt_api.py`
6. CTP 相关文件体量极大且带有生成代码属性，不适合作为本轮主治理对象。
7. 仓库内已有较多历史改进文档，因此本轮必须控制范围，避免形成新的“并行计划”而不落地。

## 4. 适用范围

本计划适用于以下任务：

1. 核心模块的小批次代码质量治理。
2. WebSocket / gateway / request layer 的稳定性增强。
3. 类型标注、文档字符串、异常处理的一致性收敛。
4. 面向回归保护的测试补强和 fixture 整理。
5. 质量门禁、测试分层和改进文档的收敛。

本计划不适用于以下任务：

1. 新交易所接入或新增业务能力。
2. CTP 自动生成文件的大规模重构。
3. 需要改动外部协议、持久化格式、认证机制的大型设计变更。
4. 一次性覆盖全部 70+ 交易所实现的全面清理。

## 5. 代码质量治理总原则

后续所有代码质量改进任务，都应遵守以下原则：

1. **先审查，再改动**
   - 先读目标代码、相关测试、调用链和异常语义。
   - 对核心调用链未审清前，不直接重写。

2. **以正确性和稳定性优先**
   - 优先级顺序：正确性 > 稳定性 > 可观测性 > 契约一致性 > 可维护性 > 可读性。

3. **小批次推进**
   - 每一批次只处理一个清晰主题。
   - 推荐单位为 `1 个问题主题 + 1 到 3 个源码文件 + 必要测试文件`。

4. **不借治理之名做功能扩张**
   - 代码质量任务不新增业务接口，不偷偷变更默认行为。

5. **证据优先**
   - 每轮都要给出问题清单、改动范围、验证命令和结论。

6. **对外行为稳定**
   - 若非明确要求，不改变公开 API、参数语义和异常语义。

7. **按项目实际设置硬门禁**
   - 每批次至少执行相关最小测试集。
   - 每批次结束前必须执行 `make check`。
   - 常规提交门禁为 `pytest tests -n 8 -m "not ctp"` 或 `./run_tests.sh -p 8`。
   - 若改动触及网络/鉴权/交易所适配层，还必须补充相应 marker 的定向验证。
   - 若环境限制导致无法完成某项验证，必须显式标注 `BLOCKED`。

## 6. 问题分级标准

### P0

必须优先处理的问题：

1. 会导致错误结果、错误状态或资源泄漏的问题。
2. 被宽泛异常吞掉，导致排障困难的失败路径。
3. WebSocket / gateway 中的生命周期管理错误、重连状态错误、线程/协程竞态。
4. 交易所请求层返回契约不稳定，可能引发上层误判的问题。
5. 修改后缺乏回归保护、回归风险较高的问题。

### P1

应在本轮内分批推进的问题：

1. 核心模块复杂度过高、单文件体量过大。
2. 重复请求逻辑、重复错误处理、重复数据清洗逻辑。
3. 核心公开 API 的类型标注或 docstring 不完整。
4. 测试数据分散、fixture 复用不足、网络依赖过重。
5. 质量文档过多且状态不一致。

### P2

可作为收尾项处理的问题：

1. 局部命名与注释不清晰。
2. 非关键路径的小型重构。
3. scripts/docs 中的风格一致性整理。

## 7. 本轮范围与批次拆分

本轮建议按 4 个批次执行，每批都可独立完成、独立验证、独立回退。

### 批次 0：基线固化与任务切片

目标：

1. 固化本轮质量基线和优先级。
2. 确定首批热点文件、现有测试、验证命令和延期项。

建议处理内容：

1. 建立“热点文件清单”和“问题清单”。
2. 明确哪些模块属于本轮治理范围，哪些延期。
3. 确认提交门禁、测试分层和 `BLOCKED` 输出规则。

交付物：

1. 本计划文档。
2. 首批任务清单。
3. 统一的输出模板。

### 批次 1：WebSocket / gateway 稳定性治理

目标：

1. 优先处理最可能影响运行稳定性的核心链路。

优先文件：

1. `bt_api_py/websocket_manager.py`
2. `bt_api_py/websocket/advanced_connection_manager.py`
3. `bt_api_py/websocket/advanced_websocket_manager.py`
4. `bt_api_py/websocket/monitoring.py`
5. `bt_api_py/gateway/runtime.py`
6. `bt_api_py/gateway/process.py`

主要动作：

1. 收敛 `except Exception` 的使用范围，补充上下文日志。
2. 清晰化连接状态、关闭流程、重试流程和错误传播路径。
3. 对线程/协程边界补最小必要测试。
4. 避免监控模块自身在异常路径上继续制造静默失败。

验收标准：

1. 热点链路中最关键的宽泛异常被替换或明确记录上下文。
2. 增补覆盖重连、关闭、异常回退的最小测试。
3. 相关最小测试集和常规提交门禁通过。

### 批次 2：请求层契约与异常一致性治理

目标：

1. 收敛高频交易所请求层的错误处理、超时、返回值和异常语义。

优先文件：

1. `bt_api_py/feeds/live_binance/request_base.py`
2. `bt_api_py/feeds/live_kucoin/request_base.py`
3. `bt_api_py/feeds/live_kraken/request_base.py`
4. `bt_api_py/feeds/live_mexc/request_base.py`
5. `bt_api_py/feeds/live_hyperliquid/request_base.py`

主要动作：

1. 统一请求失败后的异常抛出和上下文封装。
2. 补齐必要的超时、响应校验和错误消息格式。
3. 对共享模式明显的请求逻辑做轻量抽取，但避免跨全仓库大重构。
4. 为公共错误分支补最小单元测试或 contract test。

验收标准：

1. 请求层关键路径不再依赖模糊返回值表示失败。
2. 相关测试能锁定超时、错误响应、空字段和解析失败场景。
3. 变更不破坏既有调用方式。

### 批次 3：核心 API、类型与文档一致性治理

目标：

1. 提升核心模块的可维护性和静态检查质量。

优先文件：

1. `bt_api_py/bt_api.py`
2. `bt_api_py/registry.py`
3. `bt_api_py/event_bus.py`
4. `bt_api_py/core/services.py`
5. `bt_api_py/core/interfaces.py`

主要动作：

1. 为公开 API 和关键内部服务补齐类型标注。
2. 统一 Google-style docstring。
3. 进一步压缩核心层中的宽泛异常。
4. 若发现单文件职责过重，仅做小范围职责切分，不做架构迁移。

验收标准：

1. 核心 API 类型提示和 docstring 明显完整化。
2. `make type-check` 在本轮涉及模块上保持稳定。
3. 相关回归测试通过。

### 批次 4：测试治理与文档收敛

目标：

1. 让后续质量迭代可以低成本复用。

建议处理内容：

1. 收敛高频 fixture 和测试数据组织方式。
2. 区分 unit / integration / network / auth / ctp 的验证责任。
3. 将质量相关文档收敛为“一个当前计划 + 若干历史报告”。
4. 整理仍待处理的 TODO 和延期问题，形成下一轮 backlog。

验收标准：

1. 测试执行方式更清晰，新增批次不依赖临时口头约定。
2. 后续迭代可以直接复用本轮模板继续推进。

## 8. 推荐执行节奏

建议按 4 周执行：

1. **第 1 周**
   - 完成批次 0。
   - 开始批次 1 的审查和第一组修复。

2. **第 2 周**
   - 完成批次 1。
   - 开始批次 2 的请求层治理。

3. **第 3 周**
   - 完成批次 2。
   - 开始批次 3 的核心 API 类型和文档一致性治理。

4. **第 4 周**
   - 完成批次 3。
   - 执行批次 4，整理 backlog、文档和下一轮入口。

如果中途发现某批次同时触及多个不相干主题，应立即拆小，不要硬推。

## 9. 本轮必须缩范围的判定信号

出现以下任一情况，就应拆分批次，而不是继续扩张：

1. 同时涉及两个以上彼此独立的系统层级，例如 WebSocket 生命周期和请求层契约。
2. 单批次需要跨越 `3` 个以上核心模块，且主调用链尚未审清。
3. 需要修改公开 API、配置约定或返回结构。
4. 需要联动修改大量历史测试，才能“证明本轮改动合理”。
5. 无法给出清晰的最小验证集合。
6. 改动开始波及 CTP 生成代码或大量交易所实现。

## 10. 验证策略

每批次都应按以下顺序验证：

1. **最小验证**
   - 运行与改动直接相关的测试文件、测试类或 marker。

2. **模块级回归**
   - 对关键调用链补跑相邻模块测试，避免局部修复把问题扩散到链路层。

3. **质量门禁**
   - `make check`

4. **提交前基线**
   - `pytest tests -n 8 -m "not ctp"` 或 `./run_tests.sh -p 8`

5. **定向补充**
   - 若改动涉及 network/auth/exchange adapter，补跑对应 marker 或目标测试文件。

验证输出必须说明：

1. 已执行命令。
2. 结果摘要。
3. 是否阻塞提交。
4. 未执行项及原因。

## 11. 成功标准

本轮迭代完成时，应至少满足以下标准：

1. WebSocket / gateway / 请求层的 P0 问题完成首轮收敛。
2. 核心模块新增一批可复用的回归测试。
3. `make check` 成为每批次固定门禁。
4. 常规提交门禁命令在本轮主要批次中可稳定执行。
5. 核心 API 的类型标注和 docstring 完整度明显提升。
6. 形成清晰的“已完成 / 延期 / BLOCKED”记录。

## 12. 明确不做的事项

本轮不做以下内容：

1. 全量交易所实现的统一重构。
2. CTP 大文件拆分与生成逻辑治理。
3. 大规模引入新框架或重做现有架构。
4. 以“清理代码”为名改变用户可见行为。

## 13. 风险与缓解措施

### 风险

1. 热点模块链路长，容易在修一个异常分支时引出其他历史问题。
2. network/auth 相关测试可能受环境影响而不稳定。
3. 宽泛异常替换后，可能暴露出之前被隐藏的真实错误。
4. 仓库内已有多份改进文档，若继续新增平行计划，执行会分散。

### 缓解措施

1. 先做审查批次，再做实施批次。
2. 每批次控制在小范围内，并先补最小测试保护。
3. 对无法在本轮收口的问题显式延期，不临时扩张范围。
4. 每轮只保留一个当前执行计划，把其他文档视作历史背景材料。

## 14. 单批次标准输出模板

后续执行本计划中的任一批次时，建议统一输出如下结构：

```text
## 本轮范围
- 本次处理的文件、模块或问题域

## 发现的问题
- P0：
- P1：
- P2：

## 本轮实施计划
- 本轮准备处理哪些项
- 哪些项暂不处理，以及原因

## 实际改动
- 改了哪些文件
- 每项改动解决了什么问题

## 风险与未处理项
- 当前仍有哪些风险
- 哪些问题被明确延期

## 验证结果
- 已执行的验证
- 未执行但建议补充的验证
- 是否已运行 `make check`
- 是否已运行 `pytest tests -n 8 -m "not ctp"` 或 `./run_tests.sh -p 8`
- 当前是否已满足提交条件

## 下一轮建议
- 后续最值得继续推进的方向
```

如果受环境或依赖限制无法完成关键验证，必须追加如下阻塞说明：

```text
## 阻塞说明
- 状态：BLOCKED
- 原因：{依赖外部环境 / 历史失败 / 网络鉴权受限 / 调用链未审清}
- 影响：{本轮暂时无法确认的范围}
- 已完成：{已完成的审查或局部验证}
- 下一步最小动作：{建议下一步}
```

## 15. 下一步建议

本计划落地时，建议从以下顺序启动：

1. 先对 `websocket_manager.py`、`advanced_connection_manager.py`、`gateway/runtime.py` 做审查轮。
2. 将第一批 P0 问题拆成 `2` 到 `3` 个独立小任务。
3. 每个小任务只处理一个主题，并附带最小回归测试。
4. 第一批任务跑通后，再进入请求层契约治理。
