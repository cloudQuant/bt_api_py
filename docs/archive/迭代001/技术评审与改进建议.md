## 2. 原方案的主要技术缺口

### 2.1 缺口一:没有定义"插件契约" (Critical)

原文档第 12 节仅用自然语言描述插件入口形如 `register_plugin()`,但:

- 没有定义 `register_plugin()` 的函数签名 (参数、返回值、异常)。
- 没有定义插件元信息 (plugin metadata) 的数据结构。
- 没有定义核心版本与插件版本的兼容规则 (semver? caret ranges?)。
- 没有定义"插件加载失败"时的回退行为 (warn? skip? abort?)。

**影响**:Binance 团队和 OKX 团队同时开工时,无法确定各自 `plugin.py` 是否写得一致;后续接入 CTP 时会再次返工。

**建议**:在进入研发之前,必须由架构师产出一份单独的 `插件契约规范.md`,见下文第 4 章。

### 2.2 缺口二:没有"当前代码 → 目标包"的文件级映射 (High)

原文档第 8.2、11 节描述了目标包应包含什么"职责",但未给出具体文件映射表。

以 Binance 为例,当前代码散落在至少 4 处:

| 当前路径 | 行数 | 迁移去向 |
|---|---:|---|
| `bt_api_py/feeds/live_binance/` | ~7,972 | `packages/bt_api_binance/src/bt_api_binance/feeds/` |
| `bt_api_py/feeds/live_binance_feed.py` | 60 | 迁移或保留为 re-export |
| `bt_api_py/containers/exchanges/binance_exchange_data.py` | ? | `packages/bt_api_binance/src/bt_api_binance/exchange_data/` |
| `bt_api_py/exchange_registers/register_binance.py` | 92 | `packages/bt_api_binance/src/bt_api_binance/plugin.py` |
| `bt_api_py/gateway/adapters/binance_adapter.py` | ? | `packages/bt_api_binance/src/bt_api_binance/gateway/adapter.py` |
| `tests/**/test_binance_*` | ? | `packages/bt_api_binance/tests/` |

**影响**:研发很容易漏迁、错迁,或在迁移过程中引入循环依赖。

**建议**:必须产出 `文件迁移映射表.md`,精确到每个 `.py` 文件。

### 2.3 缺口三:没有识别"超大文件"的拆分策略 (High)

通过代码扫描发现:

- `bt_api_py/feeds/live_binance/request_base.py` 约 **94,289 行** (单文件),严重超出合理规模。
- `bt_api_py/feeds/live_okx/` 目录合计约 **20,349 行**。
- `bt_api_py/containers/` 目录合计约 **51,968 行**。

原方案完全没提到这些超大文件的处理。如果直接整文件搬运,插件包体积和 git 历史会异常庞大,代码评审也会极度困难。

**建议**:

1. 本轮迭代 **不拆** 这些大文件的内部结构,仅做"整文件搬运"以控制风险。
2. 产出 `packages/bt_api_binance` 时,使用 `git mv` 而非 `cp`,保留历史。
3. 把"超大文件内部按功能拆分"列为后续波次的独立技术债任务。

### 2.4 缺口四:没有定义"核心包 vs 插件包"的反向依赖禁令 (High)

原方案第 9.1、9.2 节定义了 "主包保留什么"、"插件带走什么",但没有禁令级别的强约束:

> **主包代码严禁 `import bt_api_binance` / `import bt_api_okx`**

如果没有这条硬性禁令,拆完之后很容易在主包某个"工具函数"里残留一行 `from bt_api_py.binance import ...`,导致:

- 核心包无法在只装核心时启动 (ImportError)。
- 试点插件失去可独立发布的意义。

**建议**:

1. 在 CI 中加入静态检查 (grep-based lint):主包代码不允许 import 任何 `bt_api_<exchange>` 命名空间。
2. 在 `pyproject.toml` 或 `ruff` 配置中加入 `banned-imports` 规则。
3. 写入开发者文档,作为代码评审的硬性拒审项。

### 2.5 缺口五:测试迁移策略缺失 (Medium)

原文档第 18 节讨论"测试要什么",但没讨论"现有测试怎么迁"。当前 `tests/` 下有大量 `test_binance_*.py`、`test_okx_*.py`、`tests/feeds/live_binance/**`、`tests/feeds/live_okx/**`:

- 这些测试是应该跟随插件包迁走,还是保留在核心仓库?
- 迁走后,核心仓库 CI 还能运行哪些"集成级"冒烟测试?
- 插件包的测试如何 import 核心包的测试 fixture (如 mock queue、event bus stub)?

**建议**:

1. 以"测试跟着被测代码走"为默认原则:`test_binance_*` 迁到 `packages/bt_api_binance/tests/`。
2. 在核心仓库保留 `tests/integration/test_plugin_discovery.py`,专门验证插件机制本身。
3. 把公共 fixture 提炼到 `bt_api_py/testing/` 子包 (或单独的 `bt_api_py-testkit`),插件测试可 import。

### 2.6 缺口六:Gateway adapter 的拆分顺序与 ExchangeRegistry 的拆分顺序没有排好 (Medium)

原文档隐含假设两者并行推进,但实际上存在依赖:

```
bt_api_binance/plugin.py 要同时向:
  - bt_api_py.registry.ExchangeRegistry 注册 feed/stream/balance
  - bt_api_py.gateway.runtime.GatewayRuntime 注册 adapter
```

如果 `GatewayRuntime` 的 `ADAPTER_REGISTRY` 还是硬编码 `{"BINANCE": BinanceGatewayAdapter}`,那么插件包内的 adapter 就无法被调用,而会被主包里残留的 adapter 覆盖。

**建议**:拆分必须遵守顺序:

```
Step 1: 先把 GatewayRuntime 改为可动态注册 (删除 ADAPTER_REGISTRY 硬编码)
Step 2: 在主包里保留 Binance/OKX adapter,但改为"通过插件机制注册",此时插件尚未存在
Step 3: 新建 packages/bt_api_binance,迁入 adapter,验证注册生效
Step 4: 从主包移除 Binance adapter
```

**切勿** 在第 1 步没完成前就开始搬代码。

### 2.7 缺口七:"CTP 原生扩展晚拆"的过渡期风险没定 (Medium)

原方案第 10、19.1 节把 CTP 推迟到后续波次,但没说清楚:**在 CTP 仍在主包里的这段时间,核心包的发布物还会不会继续编译 CTP?**

两种路径各有代价:

- **路径 A**:迭代 001 发布的核心包仍然包含 CTP 编译,只是新增了插件机制。
  - 优点:对 CTP 用户零感知。
  - 缺点:安装体积没下降,目标 4.1 节的"安装体验收敛"没拿到。

- **路径 B**:迭代 001 就把 CTP 从核心包 `setup.py` 中剥离,只保留代码但不打包,等后续波次再正式发布 `bt_api_ctp`。
  - 优点:安装体积立即下降。
  - 缺点:CTP 用户本轮无法用 pip 装 CTP,只能从源码构建。

**建议**:采用**路径 A**,并在迭代 001 的 release note 中明确"CTP 安装体积将在后续波次下降",给用户预期管理。

### 2.8 缺口八:版本兼容矩阵不够硬 (Low)

原方案第 20.1 节提"主版本对齐",但未给出兼容性校验的技术手段:

- 插件声明"需要核心 `>=2.0,<3.0`",这个范围写在哪里? `pyproject.toml` 的 `dependencies` 还是插件元信息?
- 核心加载插件时,如何检查插件真的兼容?
- 如果用户装了旧核心 + 新插件,报错信息长什么样?

**建议**:

1. 兼容性同时通过两种手段声明:
   - `pyproject.toml install_requires`:`bt_api_py>=2.0,<3.0` (硬阻断,pip 层面拒装)
   - 插件元信息 `core_requires = ">=2.0,<3.0"` (软校验,运行时给出清晰错误)
2. 核心加载器使用 `packaging.specifiers.SpecifierSet` 做校验,拒绝不兼容插件,但不 abort 主包启动。

## 4. 必须在开发前补齐的三份设计文档

作为技术经理,我判断**在第一行代码之前**,必须产出以下三份文档,否则研发会返工:

### 文档 A:《插件契约规范》(架构师负责)

至少包含:

1. 插件入口函数签名 (`def register_plugin(registry: ExchangeRegistry, runtime: GatewayRuntime) -> PluginInfo`)。
2. `PluginInfo` 数据类定义 (name, version, core_requires, supported_exchanges, supported_asset_types)。

5. 插件日志的统一 prefix 规范 (`[plugin:bt_api_binance]`)。
6. Python 版本与 `importlib.metadata` API 兼容策略 (3.9+ 推荐,3.8 用 `importlib_metadata` backport)。

### 文档 B:《文件迁移映射表》(主力研发负责)

以 Binance 和 OKX 为样本,逐文件列出:

| `bt_api_py/feeds/live_binance_feed.py` | 60 | 保留 | 主包保留为 shim,`from bt_api_binance.feeds import ...` | rewrite | — |
| ... | | | | | |

### 文档 C:《迁移验收清单》(QA 与研发共同负责)

包括:

1. 自动化脚本:`scripts/verify_plugin.sh`,能在干净 venv 中完成 "安装核心 → 安装插件 → 运行 smoke test"。
2. 静态检查脚本:`scripts/check_core_no_plugin_import.py`,grep 核心包不含 `bt_api_<exchange>` 导入。

## 5. 对迭代 001 范围的调整建议

原方案把 `bt_api_binance` + `bt_api_okx` 都设为 P0 并行,作为技术经理我**建议顺序调整**:

### 建议的执行顺序

| 阶段 | 内容 | 理由 |
|---|---|---|
| S1 | 核心插件协议 + 加载器 + CI 禁令 | 定契约,否则两个插件会偏离 |
| S2 | `GatewayRuntime` 去硬编码化 | 必须先于 adapter 迁移 |
| S3 | `bt_api_binance` **一个插件完整拆完** | 当作"金标样板",把坑都踩一遍 |
| S4 | `bt_api_okx` 拆分 | 此时有样板参考,速度会更快,研发投入可减半 |
| S5 | 文档、迁移指南、发布模板 | 基于真实拆分经验而非猜想 |

**关键理由**:两个插件**并行**拆分,会出现"Binance 组和 OKX 组对契约的理解不一致",最终要返工统一。串行拆分第一个后,第二个几乎是"复印样板",总工时反而更低。

### 范围建议

建议 **降低 OKX 在本轮的正式承诺等级**:

- `bt_api_binance`:P0,本轮必达,作为唯一正式发布样板。
- `bt_api_okx`:P1,本轮完成目录搭建与至少 50% 代码搬运,允许延续到迭代 002 的前 1 周收尾。

这样可以把本轮时间更多投入到插件契约稳定性上,而不是两个插件的并行协调。

## 6. 工程度量:本轮该用什么指标看是否做好了

技术经理视角,迭代 001 成功的可量化指标:

1. **安装体积**:`pip install bt_api_py` (不装 binance 插件) 比迭代 001 之前的 wheel 体积**不增加**。
2. **导入时间**:`python -c "import bt_api_py"` 在干净环境下耗时不超过当前基线的 1.2 倍。
3. **核心包导入检查**:`grep -rnE "from bt_api_binance|import bt_api_binance" bt_api_py/` 必须返回 **0** 条。
4. **插件加载可观测性**:启动时 log 中必须有明确的 `[plugin] loaded bt_api_binance v<ver>` 条目。
5. **回滚成本**:如果插件加载失败,主包仍可导入并报出清晰错误,不得抛 ImportError 穿透到用户代码。
6. **测试分布**:本轮结束后,`tests/feeds/live_binance/` 必须**全部搬到** `packages/bt_api_binance/tests/`,主仓不保留 Binance 单元测试。

## 附:交易所总盘点 (新增)

盘点主仓后确认:当前仓库共包含 **72 个 feed 目录 + 71 个 register 文件**,全部拆包工作远不止迭代 001 的 Binance/OKX。完整清单与分波次路线图见《研发TODO清单.md》阶段 5 (Wave 2~7)。

摘要分层:

| 层级 | 数量 | 说明 |
|---|---:|---|
| Tier A 头部 CEX | 17 | Binance/OKX 已在迭代001, 其余 17 个进 Wave 2 |
| Tier B 中型 CEX | 14 | Wave 3;含 4 个"孤立 feed"(无 register)需产品先确认 |
| Tier C 区域/小型 CEX | 32 | Wave 6,流水线批量处理,需产品先做用户使用度盘点 |
| Tier D DEX (web3) | 9 | Wave 4,引入 web3.py / solana-py 新依赖栈 |
| Tier E 特殊依赖 | 4 | Wave 5,CTP/IB/IB Web/MT5 单独处理 |

### 影响评估

- 原方案第 14.3 节只提到"Phase 3 拆 IB Web/MT5、Phase 4 拆 CTP、Phase 5 主包收口",**未覆盖其余 60+ 交易所**。
- 本次补全后,完整拆包专项规模是 **7 个迭代**,不是原方案想象的 5 个 Phase。
- 建议 PM / 产品侧在 Wave 3 和 Wave 6 启动前,对"孤立 feed"(ascendex/aivora/allinx/4e) 与 Tier C 小型交易所做一次用户使用度盘点,**无用户的直接废弃,不拆包**,以免把工程量做大到没必要。

### 对迭代 001 的影响

迭代 001 的承诺范围**保持不变**(只做核心插件机制 + Binance + OKX)。但必须注意:

1. 插件契约(《研发设计文档.md》§3)必须面向所有 72 个交易所设计,**不能只适配 Binance/OKX**。
2. `PluginInfo.supported_asset_types` 字段要能表达 DEX 的特殊 asset type (如 `AMM_POOL`)。
3. `GatewayRuntimeRegistrar` 要预留 CTP 的 `adapter_factory` 回调接口(因为 CTP 构造参数与其他交易所不同)。
4. CI 静态检查的"禁止反向 import"规则要覆盖所有 `bt_api_<exchange>` 命名空间,不能只针对 Binance/OKX 硬编码。
