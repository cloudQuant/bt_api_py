# 迭代 1：完善交易所接口

## 1. 背景

bt_api_py 是一个统一多交易所交易 API 框架。目前 **Binance、OKX、IB、CTP** 4 个交易所已稳定运行，其余 **64 个交易所**（58 个 CEX + 6 个 DEX/DeFi 有独立 feed 目录）虽然已有初步骨架代码，但普遍存在以下问题：

- YAML 配置中的 REST 端点路径过时或缺失
- `request_base.py` 缺少 `request()` 方法，或公共方法（`get_tick`、`get_depth` 等）未实现
- 容器类（ticker/orderbook/bar）的 `init_data()` 未返回 `self`，导致链式调用失败
- 数据规范化函数缺少 `symbol_name` 等标准字段
- 测试用例存在 `if...pass` 逻辑 bug、异步测试死代码、网络依赖测试未标记 `@pytest.mark.integration`

本迭代的目标是：逐个交易所完善接口实现、修复测试，使每个交易所的 `pytest` 全部通过。

## 3. 标准接口规范

每个 CEX 交易所需实现的接口，定义在 `bt_api_py/feeds/feed.py` 的 `Feed` 基类中。

### 3.1 行情接口（公共，无需 API Key）

| 方法 | 同步 | 异步 | 说明 | 优先级 |
|------|------|------|------|--------|
| `get_tick(symbol)` | ✅ | `async_get_tick` | 获取最新行情（Ticker） | **必须** |
| `get_depth(symbol, count)` | ✅ | `async_get_depth` | 获取深度/订单簿 | **必须** |
| `get_kline(symbol, period, count)` | ✅ | `async_get_kline` | 获取 K 线数据 | **必须** |
| `get_server_time()` | ✅ | — | 获取服务器时间 | 推荐 |
| `get_exchange_info()` | ✅ | — | 获取交易所信息/交易对 | 推荐 |

### 3.2 交易接口（私有，需要 API Key）

| 方法 | 同步 | 异步 | 说明 | 优先级 |
|------|------|------|------|--------|
| `make_order(symbol, vol, price, ...)` | ✅ | `async_make_order` | 下单 | **必须** |
| `cancel_order(symbol, order_id)` | ✅ | `async_cancel_order` | 撤单 | **必须** |
| `query_order(symbol, order_id)` | ✅ | `async_query_order` | 查询订单 | **必须** |
| `get_open_orders(symbol)` | ✅ | `async_get_open_orders` | 获取未完成订单 | 推荐 |

### 3.3 账户接口（私有，需要 API Key）

| 方法 | 同步 | 异步 | 说明 | 优先级 |
|------|------|------|------|--------|
| `get_account(symbol)` | ✅ | `async_get_account` | 获取账户信息 | **必须** |
| `get_balance(symbol)` | ✅ | `async_get_balance` | 获取余额 | **必须** |

### 3.4 异步接口

每个同步行情接口必须有对应的异步版本，遵循以下模式：

```python
def async_get_tick(self, symbol, extra_data=None, **kwargs):
    path, params, extra_data = self._get_tick(symbol, extra_data, **kwargs)
    self.submit(
        self.async_request(path, params=params, extra_data=extra_data),
        callback=self.async_callback,
    )
```

**异步方法列表（必须实现）：**
- `async_get_tick` / `async_get_ticker` — 异步获取行情
- `async_get_depth` — 异步获取深度
- `async_get_kline` — 异步获取 K 线
- `async_get_account` — 异步获取账户信息（私有）
- `async_get_balance` — 异步获取余额（私有）

**关键说明：**
- 异步方法通过 `self.submit()` 将协程提交到后台事件循环
- 结果通过 `self.async_callback()` 推入 `data_queue`
- `AsyncBase` 已自动检测系统代理，确保 aiohttp 与 requests 使用相同网络配置

### 3.5 WebSocket 接口（如交易所支持）

如果交易所提供 WebSocket API，应实现以下两类 WebSocket 类：

| 类型 | 类名模板 | 说明 |
|------|----------|------|
| 行情 WSS | `XxxMarketWssData` | 订阅公共行情推送（ticker、depth、kline、trade） |
| 账户 WSS | `XxxAccountWssData` | 订阅私有账户推送（订单变化、余额变化、持仓变化） |

**WebSocket 实现要求：**
- 参考 `bt_api_py/feeds/live_binance/market_wss_base.py` 和 `account_wss_base.py`
- 支持自动重连和心跳
- 数据通过 `data_queue` 推送
- 在 `register_xxx.py` 中注册 stream classes

### 3.6 实现模式

每个接口遵循 **三层方法** 模式（参考 Binance/HTX 实现）：

```
_get_tick(symbol, extra_data)       → 返回 (path, params, extra_data) 三元组
get_tick(symbol, extra_data)        → 调用 _get_tick() 后传入 self.request()，返回 RequestData
async_get_tick(symbol, extra_data)  → 调用 _get_tick() 后通过 self.submit(self.async_request(...)) 异步执行
```

**关键要求：**
- `_get_xxx()` 内部方法：只负责组装请求参数，返回 `(path, params, extra_data)` 三元组
- `get_xxx()` 公共方法：调用 `_get_xxx()` 获取三元组，再调用 `self.request(path, params, extra_data=extra_data)` 返回 `RequestData`
- `async_get_xxx()` 异步方法：调用 `_get_xxx()` 获取三元组，再通过 `self.submit(self.async_request(...), callback=self.async_callback)` 异步执行
- `request()` 方法：每个交易所的 `request_base.py` **必须实现**，负责构建 URL、添加认证头、调用 `self.http_request()`、包装为 `RequestData` 返回
- `async_request()` 方法：每个交易所的 `request_base.py` **必须实现**，负责异步 HTTP 请求
- 容器类的 `init_data()` 方法 **必须返回 `self`**，以支持链式调用
- 规范化函数中 **必须包含 `symbol_name`** 字段

## 5. 每个交易所的处理步骤

### 步骤 1：补全 YAML 配置

1. 阅读 `docs/exchanges/<exchange>/api.md` 中的 API 端点列表
2. 联网查询该交易所官方 API 文档，确认最新端点
3. 检查 `bt_api_py/configs/<exchange>.yaml`，确保以下 REST 路径都已配置：
   - `get_server_time`
   - `get_ticker`
   - `get_orderbook`
   - `get_klines`
   - `get_trades`
   - `get_exchange_info`（或 `get_symbols`）
   - `make_order`
   - `cancel_order`
   - `query_order`（如 API 支持）
   - `get_open_orders`（如 API 支持）
   - `get_account`
   - `get_balance`
4. 如果没有 YAML 配置文件，在 exchange_data 类中设置硬编码默认值

### 步骤 2：完善源代码

1. **`request_base.py`**：确保有 `request()` 方法
2. **公共行情方法**：实现 `get_tick`、`get_depth`、`get_kline`、`get_server_time`、`get_exchange_info`
3. **私有交易方法**：实现 `make_order`、`cancel_order`、`query_order`、`get_open_orders`
4. **私有账户方法**：实现 `get_account`、`get_balance`
5. **规范化函数**：每个 `_get_xxx_normalize_function` 确保返回 `(data_list, status)` 并包含 `symbol_name`
6. **容器类**：`init_data()` 必须返回 `self`

### 步骤 3：完善测试用例

1. 每个公共接口至少一个同步测试，验证返回 `RequestData` 且数据结构正确
2. 异步接口测试标记 `@pytest.mark.integration`，使用 `try/except queue.Empty` 模式
3. 私有接口测试标记 `@pytest.mark.integration`
4. 修复常见测试 bug：
   - `if data: pass` → `if data: actual_check(data)`
   - `except: pass` 后 `var = None` → 去掉死代码
   - `ticker.status is True` → `isinstance(ticker, RequestData)`

### 步骤 4：运行验证

```bash

SKIP_LIVE_TESTS=true pytest tests/feeds/test_<exchange>.py -q --tb=short -o "addopts="

# 验收标准：0 failed
```

### 步骤 5：标记完成

在 README.md 的交易所支持列表中将该交易所标记为 ✅ 已完成。

## 6. 常见问题与修复模式

### 6.1 源代码常见 bug

| 问题 | 典型错误 | 修复方式 |
|------|----------|----------|
| 缺少 `request()` 方法 | `AttributeError: no attribute 'request'` | 在 `request_base.py` 添加 `request()` 方法 |
| `get_server_time()` 返回 tuple | `AssertionError: isinstance(tuple, RequestData)` | 改为调用 `self.request()` 返回 RequestData |
| `init_data()` 未返回 self | `AttributeError: 'NoneType' object...` | 在 `init_data()` 末尾 `return self` |
| 规范化函数缺少 symbol_name | `assert None == "BTC-USDT"` | 在规范化函数中添加 `symbol_name` 字段 |
| `base_urls.rest` 是 dict 赋给 str 变量 | URL 拼接异常 | 从 dict 中按 asset_type 提取 URL 字符串 |
| REST 端点路径过时 | HTTP 404 | 联网查询最新 API 文档，更新 YAML/代码中的路径 |
| 方法名不匹配 | `get_tick` vs `get_ticker` | 添加别名方法 |
| 双重 request 调用 | 请求发送两次 | `_get_xxx` 只返回三元组，`get_xxx` 负责调用 `request()` |

### 6.2 测试常见 bug

| 问题 | 典型代码 | 修复方式 |
|------|----------|----------|
| `if...pass` 不保护后续代码 | `if data: pass; x = data[0]` | 改为 `if data: x = data[0]; assert ...` |
| 异步测试死代码 | `except Empty: pass; data = None; assert data` | 去掉 `data = None`，将 assert 放入 try 块 |
| 网络测试未标记 | 测试直接调用真实 API | 添加 `@pytest.mark.integration` |
| 容器 init_data 未调用 | `tick.get_last_price()` 返回 None | 先调用 `tick.init_data()` |

## 8. 限制与约束

1. **参照 Binance/OKX 标准**：所有实现应参考 `bt_api_py/feeds/live_binance/` 和 `bt_api_py/feeds/live_okx/` 的模式
2. **一次只处理一个交易所**：完全完成后才能进入下一个
3. **不允许一次运行全部测试**：耗时过长，每次只运行单个交易所的测试文件
4. **源代码优先**：测试失败时先修源代码，再改测试
5. **不可削弱测试**：不允许通过删除断言或降低覆盖率来"修复"
6. **DEX 交易所后置**：DEX/DeFi 的接口模式与 CEX 差异大，优先处理 CEX
